HW3-1: Naive DQN — Static Mode [30%]

Implements the foundational DQN algorithm with two stabilization techniques: Experience Replay and Target Network, following DRL in Action Listing 3.5/3.7.

為什麼選 Static Mode？

Static mode 的格局完全固定，玩家永遠從 (0,3) 出發，本質上只是一個「記住最佳路徑」的問題，適合驗證最基礎的 DQN 是否能正確學習。

What is DQN?

DQN 用神經網路逼近 Q(s,a)：「在狀態 s 採取動作 a，未來累積報酬的期望值」，取代傳統的 Q-table（無法處理連續或高維狀態空間）。

兩大穩定技術

📦 Experience Replay

問題：相鄰的 transition 高度相關（連續走三步，三筆資料幾乎一樣），直接訓練 → overfitting 當前軌跡，破壞之前學到的知識。

解法：存進 buffer，每次隨機抽 mini-batch，打破時序相關性。

replay.append((s1, a, r, s2, done))  # size=1000
batch = random.sample(replay, 200)   # random draw

🎯 Target Network

問題：計算 Bellman target 時用同一個正在更新的網路 → 目標不斷移動，像追一個移動的靶，導致震盪甚至發散。

解法：建立一個「凍結副本」只負責出 target，每 200 步才同步。

if step % 200 == 0:
    target.load_state_dict(model.state_dict())

ε-greedy 探索衰減

一開始完全隨機探索（ε=1.0），隨訓練進行線性衰減到最小值 0.1，讓 agent 逐漸從「亂走」轉為「利用已學知識」。

\[ \varepsilon_t = \varepsilon_{\max} - \frac{(\varepsilon_{\max} - \varepsilon_{\min}) \cdot t}{T_{\max}} \]

t = 當前 epoch，T = 總 epochs = 2000，ε: 1.0 → 0.1

Bellman Update

\[ Y = r + \gamma \cdot \max_{a'} Q_{\text{target}}(s', a') \cdot (1 - \text{done}) \]

\[ \mathcal{L} = \mathbb{E}\left[\left( Q_{\text{online}}(s, a) - Y \right)^2\right] \quad \text{(MSE)} \]

HW3-2: Double DQN & Dueling DQN — Player Mode [40%]

為什麼升級到 Player Mode？

Player mode 中，Goal/Pit/Wall 位置固定，但玩家起點有 16 種可能（4×4 格）。Agent 不能只記一條路徑，必須學習「相對於目標的導航策略」。這要求網路的泛化能力，Naive DQN 的 Q-table 思維開始出現瓶頸。

Problem: Vanilla DQN 的 Overestimation

Bellman update 裡的 max 算子會系統性高估 Q 值。直覺理解：當網路有估計誤差時，max 永遠挑「估太高的那個」——就像每次猜物品重量，永遠選看起來最重的，最終偏差越來越大。

Vanilla DQN（有偏）：

\[ Y = r + \gamma \cdot \underbrace{\max_{a'} Q_{\text{target}}(s', a')}_{\text{同一個網路 select + evaluate}} \]

① Double DQN — 解耦 selection 與 evaluation

用 online 網路選動作（「我認為最好的是哪個？」），再用 target 網路評估它（「這個動作實際上值多少？」）。兩個網路的偏差方向不同，互相抵消。

\[ Y^{\text{Double}} = r + \gamma \cdot Q_{\text{target}}\!\left(s',\; \underbrace{\arg\max_{a'} Q_{\text{online}}(s', a')}_{\text{online 選動作}}\right) \]

# Vanilla DQN — same network does both
Q2_val = target(s2).max(dim=1).values

# Double DQN — decouple: online selects, target evaluates
a_star = online(s2).argmax(dim=1, keepdim=True)   # online 選
Q2_val = target(s2).gather(1, a_star).squeeze()   # target 評

② Dueling DQN — 分離 V(s) 與 A(s,a)

傳統 DQN 對每個 (狀態, 動作) 組合都要看過才能學。Dueling 把 Q 拆成兩部分：V(s)（這個格子本身的價值，與動作無關）和 A(s,a)（在這個格子，做動作 a 比平均好多少）。

\[ Q(s, a) = V(s) + \left[ A(s,a) - \frac{1}{|A|}\sum_{a'} A(s,a') \right] \]

減去 mean(A) 是為了確保 V 和 A 的分解唯一（否則可以任意平移 V 和 A 得到相同 Q）。

class DuelingDQN(nn.Module):
    def forward(self, x):
        feat = self.feature(x)               # 共享特徵
        V    = self.value(feat)               # 純量 V(s)
        A    = self.advantage(feat)           # 4 維 A(s,·)
        return V + A - A.mean(dim=1, keepdim=True)  # 減 mean 保證唯一性

為什麼 Dueling 特別適合 Player Mode？

Player mode 的 Goal/Pit 位置固定。格子 (0,1) 旁邊的狀態「本身就很危險」，不管往哪走都要小心 → 這是 V(s) 的工作。格子 (0,2) 往左走比往其他方向好 → 這是 A(s,a) 的工作。兩個資訊分開學，效率更高。

HW3-3: PyTorch Lightning — Random Mode [30%]

將訓練迴圈改寫為 PyTorch Lightning LightningModule，並同時攻克最難的 random mode（每 episode 所有物件位置全部隨機）。

為什麼 Random Mode 最難？

Random mode 每個 episode 的 Goal/Pit/Wall 位置都不同。Agent 不能記路徑，必須學到抽象策略：看當前 state 的 64 維向量，理解「Goal 在哪、Pit 在哪」，然後選擇正確方向。這需要真正的泛化能力。

RL + Lightning 的關鍵設計：ReplayDataset

Lightning 預期標準的 Dataset/DataLoader，但 RL 的 replay buffer 是一個動態變化的 deque。解法：把 deque 包裝成 Dataset，每個 training step 在 training_step() 裡先收集新經驗再訓練。

class ReplayDataset(Dataset):
    def __init__(self, buffer: deque):
        self._buf = buffer

    def __getitem__(self, idx):
        s1, a, r, s2, done = self._buf[idx]
        return (s1.squeeze(0),
                torch.tensor(a,    dtype=torch.long),
                torch.tensor(r,    dtype=torch.float32),
                s2.squeeze(0),
                torch.tensor(done, dtype=torch.float32))

class DQNLightning(pl.LightningModule):
    def training_step(self, batch, batch_idx):
        self.collect_experience()   # 先收集，再從 batch 取樣訓練
        s1, a, r, s2, done = batch

        a_star = self.online(s2).argmax(dim=1, keepdim=True)
        Q2_val = self.target(s2).gather(1, a_star).squeeze()
        Y      = r + GAMMA * (1 - done) * Q2_val
        loss   = self.loss_fn(Q1_a, Y)
        self.log('train_loss', loss, prog_bar=True)
        return loss

四大訓練技術

✂️ Gradient Clipping

gradient_clip_val=1.0 — 將梯度的 L2 norm 截斷至 1.0。防止 TD error 突然爆炸導致網路權重大幅震盪，在 random mode 這種高方差環境下尤其重要。

📉 LR Scheduling

StepLR(step_size=1000, gamma=0.5) — 每 1000 epoch 學習率減半。訓練初期需要大步伐探索，後期需要小步伐精調。自動化這個過程避免手動調整。

🪄 Huber Loss

SmoothL1Loss — 小誤差時像 MSE（平滑），大誤差時像 L1（對離群值不那麼敏感）。比純 MSE 更能抵抗偶爾出現的極大 TD error。

💾 Best Checkpoint

ModelCheckpoint(monitor='win_rate_100') — 自動儲存 100-episode 滾動勝率最高時的模型，而非訓練結束的模型。這是 HW3-3 最終得到 95.8% 的關鍵。

trainer = pl.Trainer(
    max_epochs=5000,
    gradient_clip_val=1.0,          # 技術 1
    accelerator='gpu', devices=1,   # GPU 加速
    callbacks=[ModelCheckpoint(
        monitor='win_rate_100',
        mode='max', save_top_k=1,    # 技術 4：存最佳
    )],
)
# configure_optimizers 內加 StepLR（技術 2）

🌈 HW3-4: Rainbow DQN — Random Mode (Bonus)

Rainbow (Hessel et al., 2018) combines multiple independent improvements into one agent. We implement 4 core components on the hardest random mode, then analyze why results fell short — and build an improved version.

① Double DQN ② Dueling Network ③ Prioritized Replay ④ N-step Returns

③ Prioritized Experience Replay (PER)

Standard replay samples uniformly — wasting compute on transitions the agent already handles well. PER samples proportional to TD error: surprising transitions are revisited more often.

\[ P(i) = \frac{p_i^\alpha}{\sum_k p_k^\alpha}, \quad p_i = |\delta_i| + \varepsilon \]

\[ w_i = \left(\frac{1}{N \cdot P(i)}\right)^\beta \quad (\text{importance-sampling correction}) \]

Implemented with a 1-indexed SumTree for O(log n) priority sampling (root at index 1, leaves at [capacity, 2×capacity−1]):

# 1-indexed SumTree: root=1, children of i → 2i, 2i+1
class SumTree:
    def get(self, s):
        idx      = self._retrieve(1, s)     # start from root=1
        data_idx = idx - self.capacity
        return idx, self.tree[idx], self.data[data_idx]

# Importance-weighted Huber loss
loss = (w_tensor * smooth_l1_loss(X, Y, reduction='none')).mean()

④ N-step Returns

Instead of bootstrapping after 1 step, accumulate n=2 steps of real reward before bootstrapping. This reduces variance while keeping bias low on the small 4×4 grid.

\[ Y_n = \sum_{k=0}^{n-1} \gamma^k r_{t+k} + \gamma^n \max_a Q(s_{t+n}, a) \]

class NStepBuffer:
    def get(self):
        if len(self.buf) < self.n: return None
        R = 0.0
        for i, (_, _, r, _, d) in enumerate(self.buf):
            R += (self.gamma ** i) * r
            if d: break               # stop at terminal
        sn = self.buf[-1][3]
        self.buf.pop(0)
        return (s0, a0, R, sn, done_n)

問題分析：為什麼 Greedy Eval 只有 23%？

原版 Rainbow 訓練途中滾動 win rate 峰值達 67.7%，但最終 greedy eval 僅 23%。三個原因：

📉 無 Checkpoint 保存

訓練結束時使用「最後的模型」，而非「最好的模型」。Q-values 在長訓練後持續漂移，peak 67.7% 的權重已被覆蓋。

🧩 State 表示效率低

64-dim one-hot 迫使網路從 4 個 4×4 位元矩陣中自己學出「距離」概念。泛化到新的隨機佈局特別困難。

🔀 無課程學習

直接面對最難的 random mode（全隨機佈局），沒有從簡單場景建立基礎策略，前期探索效率極低。
