HW3-1: Naive DQN — Static Mode [30%]

Implements the foundational DQN algorithm with two stabilization techniques: Experience Replay and Target Network, following DRL in Action Listing 3.5/3.7.

What is DQN?

DQN approximates Q(s,a) — "how much total future reward will I get if I take action a in state s?" — with a neural network, instead of a lookup table.

Key Techniques

📦 Experience Replay

Store transitions (s, a, r, s', done) in a circular buffer (size 1000). Sample random mini-batches of 200 to break temporal correlations and improve sample efficiency.

replay.append((s1, a, r, s2, done))
batch = random.sample(replay, 200)

🎯 Target Network

A frozen copy of the online network used to compute Bellman targets. Updated every 200 steps. Prevents the "moving target" problem that causes training oscillation.

if step % sync_freq == 0:
    target.load_state_dict(
        model.state_dict())

Bellman Update (Training Loss)

\[ Y = r + \gamma \cdot \max_{a'} Q_{\text{target}}(s', a') \cdot (1 - \text{done}) \]

\[ \mathcal{L} = \left( Q_{\text{online}}(s, a) - Y \right)^2 \]

Training Loop (Simplified)

for epoch in range(2000):
    game = Gridworld(mode='static')

        # Mini-batch Bellman update

        Y = r + γ * (1-done) * max(target(s2))
        loss = MSE(online(s1)[a], Y)
        loss.backward()

HW3-2: Double DQN & Dueling DQN — Player Mode [40%]

Problem: Overestimation in Vanilla DQN

The max operator in the Bellman update causes systematic overestimation of Q-values. When the network is noisy, max always picks the overestimated value, leading to divergence or suboptimal policies.

Vanilla DQN target (biased):

\[ Y = r + \gamma \cdot \underbrace{\max_{a'} Q_{\text{target}}(s', a')}_{\text{overestimates!}} \]

① Double DQN

Decouple action selection (which action is best?) from action evaluation (how good is it?):

\[ Y^{\text{Double}} = r + \gamma \cdot Q_{\text{target}}\!\left(s',\; \underbrace{\arg\max_{a'} Q_{\text{online}}(s', a')}_{\text{online selects}}\right) \]

# Vanilla DQN
Q2_val = target(s2).max(dim=1).values

# Double DQN — two lines change
a_star = online(s2).argmax(dim=1, keepdim=True)   # online selects
Q2_val = target(s2).gather(1, a_star).squeeze()   # target evaluates

② Dueling DQN

Split the Q-value into two separate streams: state Value V(s) and action Advantage A(s,a).

\[ Q(s, a) = V(s) + \left[ A(s,a) - \frac{1}{|A|}\sum_{a'} A(s,a') \right] \]

Mean subtraction makes V and A identifiable (unique decomposition).

Q = V + A − mean(A)
class DuelingDQN(nn.Module):
    def forward(self, x):
        feat = self.feature(x)          # shared layers
        V    = self.value(feat)           # scalar V(s)
        A    = self.advantage(feat)       # vector A(s,·)
        return V + A - A.mean(dim=1, keepdim=True)

Why Dueling Helps for Player Mode

In player mode, the goal/pit positions are fixed. Many states near the goal are intrinsically valuable regardless of which action is taken. The Value stream V(s) learns this directly — without needing to observe every (state, action) pair explicitly.

Training Techniques

✂️ Gradient Clipping

Prevents exploding gradients. Trainer(gradient_clip_val=1.0) clips gradient norm to 1.0 each step.

📉 LR Scheduling

StepLR(step_size=1000, gamma=0.5): halves the learning rate every 1000 epochs as training stabilizes.

🪄 Huber Loss

SmoothL1Loss — behaves like MSE for small errors and L1 for large ones. Robust to outlier TD errors.

💾 Checkpointing

ModelCheckpoint saves the best model by win rate. Easy to resume training or deploy best weights.

class DQNLightning(pl.LightningModule):
    def training_step(self, batch, idx):
        # Collect new experience each step
        self.collect_experience()
        # Double DQN Bellman update
        a_star = self.online(s2).argmax(dim=1, keepdim=True)
        Q2_val = self.target(s2).gather(1, a_star).squeeze()
        Y      = r + GAMMA * (1 - done) * Q2_val
        loss   = self.loss_fn(Q1_a, Y)
        self.log('train_loss', loss, prog_bar=True)
        return loss

    def configure_optimizers(self):
        opt  = Adam(self.online.parameters(), lr=1e-3)
        sch  = StepLR(opt, step_size=1000, gamma=0.5)
        return {"optimizer": opt, "lr_scheduler": {"scheduler": sch}}

trainer = pl.Trainer(
    max_epochs=5000,
    gradient_clip_val=1.0,    # ← training technique
    callbacks=[ModelCheckpoint(monitor='win_rate_100')],

🌈 HW3-4: Rainbow DQN — Random Mode (Bonus)

Rainbow (Hessel et al., 2018) combines multiple independent improvements into one agent, achieving significantly better sample efficiency than any single improvement alone.

① Double DQN ② Dueling Network ③ Prioritized Replay ④ N-step Returns

③ Prioritized Experience Replay (PER)

Standard replay samples uniformly — wasting compute on transitions the agent already handles well. PER samples proportional to TD error: surprising transitions are revisited more often.

\[ P(i) = \frac{p_i^\alpha}{\sum_k p_k^\alpha}, \quad p_i = |\delta_i| + \varepsilon \]

\[ w_i = \left(\frac{1}{N \cdot P(i)}\right)^\beta \quad (\text{importance-sampling correction}) \]

Implemented with a SumTree for O(log n) priority sampling:

# SumTree: internal nodes hold sums, leaves hold priorities
class SumTree:
    def get(self, s):          # sample by value s
        idx = self._retrieve(0, s)
        return idx, self.tree[idx], self.data[idx - self.capacity + 1]

# In training loop: weight the loss by importance-sampling weight
loss = (w_tensor * smooth_l1_loss(X, Y, reduction='none')).mean()

④ N-step Returns

Instead of bootstrapping after 1 step, accumulate n=3 steps of real reward before bootstrapping. This provides more accurate credit assignment and reduces variance.

\[ Y_n = \sum_{k=0}^{n-1} \gamma^k r_{t+k} + \gamma^n \max_a Q(s_{t+n}, a) \]

class NStepBuffer:
    def get(self):
        if len(self.buf) < self.n: return None
        R = sum(γ**i * r for i,(s,a,r,s2,d) in enumerate(self.buf))
        s0, a0 = self.buf[0][:2]
        sn = self.buf[-1][3]
        return (s0, a0, R, sn, any_done)
